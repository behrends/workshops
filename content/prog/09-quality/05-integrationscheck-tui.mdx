# Einfacher Integrationscheck im TUI

<Callout>
  **Dauer:** 35 Minuten

  **Ziel:** Einen kritischen TUI-Ablauf als kleinen Integrationscheck
  mit 1 Erfolgs- und 1 Fehlerfall testen
</Callout>

In der [vorherigen Lektion](/prog/09-quality/04-unit-testing-vitest) haben
wir einzelne Funktionen isoliert mit Unit-Tests geprüft. Jetzt gehen wir
einen Schritt weiter: Bei einem Integrationscheck testen wir das
Zusammenspiel mehrerer Teile:

- Eingabe aufbereiten
- Eingaben validieren
- Daten speichern

Für TUI-Apps ist das meist robuster als UI-Automation auf Zeichenebene.
Wir testen den fachlichen Ablauf direkt über einen Use Case.

## Beispiel-Ablauf

Kritischer Ablauf in unserer Wetter-App: „Stadt als Favorit speichern“

1. Nutzereingabe wird übernommen (`rawInput`)
2. Eingabe wird validiert
3. Bei gültiger Eingabe wird gespeichert

Dateistruktur:

```txt
src/
  favorites/
    add-favorite-flow.js
    favorites-repo-memory.js
    add-favorite-flow.integration.test.js
```

<Callout type="info">
  Die Beispiele hier verwenden `.js`-Dateien. Dank `allowJs: true` in der
  `tsconfig.json` funktioniert das im selben Projekt neben `.ts`-Dateien.
  TypeScript kann so schrittweise eingeführt werden.
</Callout>

## Produktcode

`src/favorites/add-favorite-flow.js`:

```js
function validateCity(input) {
  const city = input?.city?.trim();

  if (!city) {
    throw new Error('Stadtname ist erforderlich');
  }

  return { city };
}

export async function addFavoriteFlow(rawInput, repo) {
  const validated = validateCity(rawInput);
  const saved = await repo.save(validated);
  return saved;
}
```

`src/favorites/favorites-repo-memory.js`:

```js
export function createMemoryFavoritesRepo() {
  const favorites = [];
  let nextId = 1;

  return {
    async save(favorite) {
      const saved = { id: nextId++, ...favorite };
      favorites.push(saved);
      return saved;
    },
    async all() {
      return favorites;
    },
  };
}
```

## Integrationscheck mit Vitest

`src/favorites/add-favorite-flow.integration.test.js`:

```js
import { describe, expect, it } from 'vitest';
import { addFavoriteFlow } from './add-favorite-flow.js';
import { createMemoryFavoritesRepo } from './favorites-repo-memory.js';

describe('addFavoriteFlow (Integration)', () => {
  it('Erfolgsfall: validiert und speichert Favorit', async () => {
    const repo = createMemoryFavoritesRepo();

    const saved = await addFavoriteFlow({ city: 'Berlin' }, repo);

    expect(saved.id).toBe(1);
    expect(saved.city).toBe('Berlin');
    expect(await repo.all()).toHaveLength(1);
  });

  it('Fehlerfall: leerer Stadtname wird nicht gespeichert', async () => {
    const repo = createMemoryFavoritesRepo();

    await expect(
      addFavoriteFlow({ city: '' }, repo),
    ).rejects.toThrow('Stadtname ist erforderlich');

    expect(await repo.all()).toHaveLength(0);
  });
});
```

Ausführen:

```bash
npm run test
```

## Einbindung in die TUI

Die TUI ruft denselben Use Case auf, den auch der Integrationscheck testet:

```js
import { addFavoriteFlow } from './favorites/add-favorite-flow.js';

async function onAddFavorite(formValues, repo) {
  try {
    await addFavoriteFlow(formValues, repo);
    return 'Stadt wurde als Favorit gespeichert';
  } catch (error) {
    return error.message;
  }
}
```

Dadurch prüfen die Integrationstests genau den Codepfad, den die App
im Alltag ebenfalls nutzt.

<Callout type="info">
  **Praxisregel für dieses Projekt**

  Wählt pro Feature mindestens einen kritischen Ablauf und schreibt dafür
  genau zwei Integrationschecks:

  - 1 Erfolgsfall
  - 1 Fehlerfall
</Callout>
