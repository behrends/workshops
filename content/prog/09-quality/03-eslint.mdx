# Linting mit ESLint

<Callout>
  **Dauer:** 45 Minuten

  **Ziel:** Automatisierte Qualitätsregeln für Code im Team etablieren
</Callout>

[ESLint](https://eslint.org/) überprüft Code auf typische Probleme:

- mögliche Fehler (z. B. undefinierte Variablen),
- wartbarkeitsrelevante Muster,
- inkonsistente Schreibweisen.

Gerade bei KI-generiertem Code hilft ein Linter, problematische Stellen
früh und reproduzierbar zu erkennen.

## Was ist ein Linter?

Ein Linter analysiert Quellcode statisch und meldet typische Fehler,
problematische Muster und Wartbarkeitsrisiken. Im Unterschied zu einem
Formatter wie Prettier geht es beim Linting nicht primär um Einrückung
oder Zeilenumbrüche, sondern um inhaltliche Codequalität.

## Minimal-Setup (Node.js, modernes ESLint)

```bash
npm install -D eslint @eslint/js globals
```

Optional kann die Konfiguration auch per Initialisierungsbefehl erzeugt werden:

```bash
npm init @eslint/config@latest
```

Dieser Befehl installiert ESLint und erstellt eine Konfigurationsdatei
`eslint.config.js`.

Beispiel für `eslint.config.js`:

```js
import js from '@eslint/js';
import globals from 'globals';

export default [
  js.configs.recommended,
  {
    files: ['**/*.js'],
    languageOptions: {
      sourceType: 'module',
      globals: {
        ...globals.node,
      },
    },
    rules: {
      'no-unused-vars': ['warn', { argsIgnorePattern: '^_' }],
      'no-console': 'off',
      eqeqeq: ['error', 'always'],
    },
  },
];
```

npm-Scripts:

```json
{
  "scripts": {
    "lint": "eslint .",
    "lint:fix": "eslint . --fix"
  }
}
```

## Editor-Integration (VS Code)

Die offizielle [ESLint-Extension für VS Code](https://marketplace.visualstudio.com/items?itemName=dbaeumer.vscode-eslint)
markiert Probleme direkt im Editor und zeigt sie zusätzlich in der
Problems-Ansicht.

## Beispiel: Was ESLint meldet

```js
function calcScore(points, bonus) {
  const total = points + bonus;
  if (total == 10) {
    console.log("Treffer");
  }
  return points;
}
```

Typische Hinweise:

- `eqeqeq`: `==` soll durch `===` ersetzt werden
- `no-unused-vars`: `total` wird eventuell nicht sinnvoll weiterverwendet

## Typische Regeln im Alltag

- `no-unused-vars`: meldet ungenutzte Variablen oder Parameter
- `eqeqeq`: erzwingt strikte Vergleiche (`===` statt `==`)
- `no-undef`: meldet nicht deklarierte Variablen

Eine vollständige Übersicht gibt die ESLint-Dokumentation:
[Rules Reference](https://eslint.org/docs/latest/rules/).

## Fehlergrade

Jede ESLint-Regel kann mit einem Fehlergrad konfiguriert werden:

- `"off"` (oder `0`): Regel deaktiviert
- `"warn"` (oder `1`): Warnung
- `"error"` (oder `2`): Fehler (Exit-Code ungleich 0)

## Zusammenspiel mit Prettier

ESLint und Prettier sollten klar getrennte Aufgaben übernehmen:

- Prettier formatiert den Code
- ESLint prüft Codequalität und potenzielle Fehler

Damit es keine Konflikte bei Formatierungsregeln gibt, kann
`eslint-config-prettier` ergänzt werden:

```bash
npm install --save-dev eslint-config-prettier
```

Beispiel:

```js
import js from '@eslint/js';
import globals from 'globals';
import prettier from 'eslint-config-prettier';

export default [
  js.configs.recommended,
  prettier,
  {
    files: ['**/*.js'],
    languageOptions: {
      sourceType: 'module',
      globals: {
        ...globals.node,
      },
    },
    rules: {
      'no-unused-vars': ['warn', { argsIgnorePattern: '^_' }],
      eqeqeq: ['error', 'always'],
    },
  },
];
```
